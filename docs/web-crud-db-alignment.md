# Web CRUD vs Backend/DB Alignment

## 1. Назначение документа
Этот документ фиксирует расхождения между ожидаемыми CRUD-сценариями веб-интерфейса и тем, как сейчас устроены backend API, схемы и база данных.

Документ нужен, чтобы:
- прорабатывать расхождения по одному;
- не потерять блокеры для frontend-реализации;
- отделить уже согласованные сценарии от тех, которые требуют изменения backend, документации или UX.

Связанные документы:
- `README.md`
- `docs/PRD.md`
- `docs/frontend-architecture.md`
- `docs/backend-architecture.md`
- `docs/api-contract.md`
- `docs/permission-matrix.md`

## 2. Статус
Текущее состояние:
- backend уже реализует значительную часть CRUD-операций;
- часть веб-сценариев уже совпадает с backend и БД;
- часть сценариев упирается в реальные ограничения схем, таблиц и permission logic.

## 3. Ключевые расхождения

### 3.1 Создание питомца сейчас требует фото как обязательное поле
Чеклист реализации:
- [x] Зафиксировать продуктовое решение: фото необязательно при создании питомца.
- [x] Сделать `pet_photo_object_key` необязательным в API-схеме создания питомца.
- [x] Привести `PetResponse` к модели, допускающей отсутствие фото.
- [x] Разрешить `NULL` для `pet_photo_object_key` на уровне ORM-модели.
- [x] Подготовить миграцию БД для снятия `NOT NULL` и перевода пустых строк в `NULL`.
- [x] Унифицировать backend-логику вокруг `NULL` как состояния "фото отсутствует".
- [x] Обновить рабочую и API-документацию под новый контракт.

Проблема:
- веб-сценарий описывает создание карточки питомца как отдельный основной flow;
- фото питомца в продуктовой модели выглядит как опциональная часть профиля и поддерживается отдельным upload flow;
- backend-схема `PetCreate` требует `pet_photo_object_key`;
- таблица `pets_info` тоже хранит `pet_photo_object_key` как обязательное поле.

Следствие:
- веб не сможет создать питомца без предварительной загрузки фото;
- onboarding "сначала создать питомца, потом при желании добавить фото" сейчас не совпадает с backend.

Что нужно проработать:
- сделать ли `pet_photo_object_key` необязательным на уровне схемы и БД;
- или зафиксировать, что фото обязательно уже при создании питомца;
- или разрешить создание питомца с пустым значением фото как технический компромисс.

Решение:
- фото питомца является необязательным шагом;
- пользователь может пропустить загрузку фото при создании питомца;
- если фото не загружено, карточка питомца все равно должна успешно создаваться;
- если шаг загрузки фото пропущен, значения, связанные с фотографией питомца, должны оставаться пустыми.

Целевое поведение:
- `POST /pets` должен принимать создание питомца без `pet_photo_object_key`;
- поля фото должны считаться необязательными и не блокировать onboarding;
- загрузка фото должна оставаться отдельным последующим действием через существующий photo upload flow;
- отсутствие фото не должно считаться ошибкой валидации ни на frontend, ни на backend.

Что это означает для backend и БД:
- `pet_photo_object_key` нужно сделать необязательным в схеме создания питомца;
- на уровне БД поле `pet_photo_object_key` должно допускать пустое значение или `NULL`;
- связанные metadata-поля фото должны оставаться пустыми до фактической загрузки изображения;
- `POST /pets` должен корректно сохранять питомца без фото и без искусственного технического значения.

### 3.2 Регистрация пользователя требует больше данных, чем ожидает базовый веб-flow
Чеклист реализации:
- [x] Зафиксировать решение: регистрация остаётся полной и расширенной.
- [x] Зафиксировать, что фото пользователя является необязательным полем.
- [x] Сделать `user_photo` необязательным в API-схемах пользователя.
- [x] Привести публичные и приватные user response-схемы к модели, допускающей отсутствие фото.
- [x] Разрешить `NULL` для `user_photo` на уровне ORM-модели.
- [x] Подготовить миграцию БД для снятия `NOT NULL` и перевода пустых строк в `NULL`.
- [x] Унифицировать backend-логику создания и обновления пользователя вокруг `NULL` как состояния "фото отсутствует".
- [x] Привести Telegram-auth flow к той же модели хранения отсутствующего фото.
- [x] Обновить документацию под новый контракт регистрации.

Проблема:
- экран входа и регистрации в документации описан как обычный auth-flow;
- backend `POST /users` требует не только имя, email и пароль, но и `phone_number`, `birth_date`, `photo_url`;
- поле `user_photo` в БД также обязательное.

Следствие:
- если веб сделает короткую регистрацию, backend не примет такие данные;
- логика "зарегистрироваться быстро, а профиль заполнить позже" сейчас не поддерживается.

Что нужно проработать:
- оставить ли расширенную регистрацию обязательной;
- или разрешить минимальную регистрацию с последующим заполнением профиля;
- или разделить account creation и profile completion на два разных этапа.

Решение:
- регистрация пользователя остаётся полной и расширенной;
- обязательными остаются имя, email, телефон, пароль и дата рождения;
- фото пользователя не является обязательным полем;
- фото может быть загружено позже или не загружаться вовсе.

Целевое поведение:
- `POST /users` должен успешно создавать пользователя без `user_photo`;
- отсутствие фото не должно считаться ошибкой валидации;
- профиль пользователя должен корректно возвращаться с пустым значением фото (`null`);
- update профиля должен позволять добавить фото позже.

Что это означает для backend и БД:
- `user_photo` нужно сделать необязательным в схеме создания пользователя;
- на уровне БД поле `user_photo` должно допускать `NULL`;
- Telegram-auth и обычная регистрация должны использовать одну и ту же модель отсутствующего фото;
- отсутствие фото не должно подменяться технической пустой строкой.

### 3.3 Для формы питомца нет публичного API справочников типов и пород
Чеклист реализации:
- [x] Зафиксировать решение: frontend получает типы животных и породы из БД.
- [x] Зафиксировать, что породы должны быть привязаны к типу животного.
- [x] Добавить связь `AnimalBreed -> AnimalType` на уровне модели данных.
- [x] Добавить backend API для списка типов животных.
- [x] Добавить backend API для получения пород по выбранному типу животного.
- [x] Привести Telegram-бот к той же модели выбора пород по типу.
- [x] Обновить документацию под новый reference-data contract.
- [x] Применить миграцию БД и проверить, что существующие породы привязаны к типам.

Проблема:
- создание и редактирование питомца требуют `animal_type_id` и `animal_breed_id`;
- в текущем API нет явных публичных маршрутов для получения списка типов животных и пород.

Следствие:
- веб-интерфейсу неоткуда брать корректные значения для select-полей;
- фронтенд может начать хардкодить справочники, что быстро приведет к рассинхронизации с базой.

Что нужно проработать:
- добавить API для `animal types`;
- добавить API для `animal breeds`;
- определить формат фильтрации пород по выбранному типу животного.

Решение:
- frontend должен получать список типов животных из БД;
- после выбора типа frontend должен получать список пород, относящихся только к этому типу;
- породы животных должны быть связаны с типом животного на уровне БД и backend-модели.

Целевое поведение:
- `GET /animal-types` возвращает список типов животных;
- `GET /animal-types/{animal_type_id}/breeds` возвращает только породы выбранного типа;
- frontend не хардкодит справочники;
- выбор породы всегда ограничен выбранным типом животного.

Что это означает для backend и БД:
- таблица `animals_breeds` должна хранить `animal_type_id`;
- backend должен обеспечивать публичный read-only доступ к этим справочникам;
- существующие записи пород нужно мигрировать и связать с типами;
- backend и bot flows должны использовать одну и ту же логику фильтрации пород по типу.

### 3.4 ИИ-чат в целевой модели pet-centric, но backend и БД пока user-centric
Проблема:
- документация frontend и PRD описывают чат по конкретному питомцу;
- текущий backend создаёт чаты только по `user_id`;
- модель `llm_chats` не содержит `pet_id`.

Следствие:
- веб-маршруты и UX для `/pets/:petId/chat` не соответствуют текущей модели данных;
- нельзя корректно хранить отдельную историю чатов по каждому питомцу;
- права `shared_user` на чат питомца тоже не могут быть реализованы в текущем виде.

Что нужно проработать:
- добавить ли `pet_id` в модель чата;
- как мигрировать текущие user-centric чаты;
- как будет выглядеть доступ `shared_user` к чатам питомца;
- какие поля и endpoints станут частью нового chat contract.

### 3.5 Update для документов формально есть, но почти не дает редактирования
Чеклист реализации:
- [x] Зафиксировать решение: пользователь может задавать `custom_name` при загрузке документа.
- [x] Зафиксировать решение: пользователь может менять `custom_name` и тип документа после загрузки.
- [x] Расширить upload request/complete request поддержкой `custom_name`.
- [x] Расширить update request поддержкой `custom_name` и реальной смены `document_type_id`.
- [x] Обновить backend-логику сохранения `custom_name` как пользовательского display name.
- [x] Обновить документацию под новый document metadata flow.

Проблема:
- веб-архитектура описывает `PATCH /pets/{pet_id}/documents/{document_row_id}`;
- текущая схема update принимает только `document_type_id`;
- сервис фактически не разрешает менять тип уже загруженного файла;
- других редактируемых метаданных у документа сейчас нет.

Следствие:
- для веба это выглядит как наличие полноценного `Update`, хотя реального редактирования почти нет;
- пользователь не сможет менять имя документа или другие свойства, если такой UX будет заложен.

Что нужно проработать:
- нужен ли документу реальный edit-сценарий;
- нужно ли поддерживать `custom_name`;
- оставить ли update только как технический endpoint;
- или убрать ожидание полноценного document edit из frontend/UI.

Решение:
- при загрузке документа пользователь может задать `custom_name`;
- если `custom_name` не задан, backend использует имя типа документа как значение по умолчанию;
- после загрузки пользователь может изменить и `custom_name`, и `document_type_id`;
- `custom_name` хранится как пользовательское отображаемое имя документа.

Целевое поведение:
- `POST /pets/{pet_id}/documents/upload-url` принимает `custom_name`;
- `POST /pets/{pet_id}/documents/complete` может принять `custom_name`, если frontend завершает upload отдельным шагом;
- `PATCH /pets/{pet_id}/documents/{document_row_id}` позволяет менять `custom_name`;
- `PATCH /pets/{pet_id}/documents/{document_row_id}` позволяет менять тип документа.

Что это означает для backend:
- document metadata flow становится полноценным с точки зрения UX;
- `custom_name` больше не должен вычисляться только из object key;
- запрет на смену типа документа после загрузки снимается;
- download flow продолжает работать через `object_key`, а UI опирается на хранимый `custom_name`.

## 4. Что уже совпадает

### 4.1 Питомцы
- владелец может создавать, читать, обновлять и удалять питомцев;
- список питомцев объединяет собственных и расшаренных питомцев;
- `shared_user` может читать карточку питомца.

### 4.2 Фото питомца
- фото уже поддерживается отдельным presigned upload flow;
- есть получение upload URL, завершение загрузки, download URL и удаление;
- доступ на изменение фото ограничен владельцем.

### 4.3 Документы питомца
- документы поддерживают list/create/download/delete;
- `shared_user` уже может читать и скачивать документы;
- владелец может загружать и удалять документы.

### 4.4 Sharing
- владелец может создавать invite;
- пользователь может принять invite;
- владелец может отозвать доступ;
- список shared users уже поддерживается backend.

## 5. Приоритет проработки

### P0
- Создание питомца без обязательного фото. Выполнено на уровне backend, БД и документации.
- Регистрация пользователя без обязательного полного профиля. Уточнено: профиль остаётся полным, но фото сделано необязательным. Выполнено на уровне backend, БД и документации.
- Справочники `animal types` и `animal breeds` для веб-форм. Выполнено на уровне backend, БД и документации.
- Переход от user-centric чата к pet-centric.

### P1
- Переосмысление update-сценария для документов.
- Финализация permission logic для `shared_user` в pet-centric chat.

### P2
- Дополнительная унификация документации после принятия решений по P0/P1.

## 6. Рекомендуемый формат проработки
Для каждого пункта стоит зафиксировать:
- целевое поведение;
- текущее ограничение;
- решение на уровне UX;
- решение на уровне API;
- решение на уровне БД;
- нужно ли изменение документации;
- нужно ли изменение миграций и существующих данных.

## 7. Следующий шаг
Самый практичный порядок разбора:
1. Создание питомца и обязательность фото. Частично проработано: продуктовое решение зафиксировано, требуется привести backend и БД в соответствие.
   Статус: backend, БД и документация синхронизированы.
2. Регистрация и обязательность пользовательского профиля.
   Статус: решение уточнено. Регистрация остаётся полной, но фото пользователя больше не обязательно; backend, БД и документация синхронизированы.
3. API справочников типов и пород.
   Статус: backend API, модель данных и миграция БД применены.
4. Pet-centric ИИ-чат.
5. Реальный update-сценарий документов.

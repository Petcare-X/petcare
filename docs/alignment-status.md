# Alignment Status: Vision vs Current Implementation

## 1. Назначение документа
Этот документ показывает, где текущее состояние проекта уже совпадает с целевым видением, где совпадение пока только частичное, а где требуются явные доработки.

Документ нужен для:
- планирования следующего этапа разработки;
- быстрой ориентации для команды и ИИ-агентов;
- снижения риска путаницы между "что уже есть" и "что мы хотим получить".

Связанные документы:
- `docs/PRD.md`
- `docs/frontend-architecture.md`
- `docs/backend-architecture.md`
- `docs/permission-matrix.md`
- `docs/api-contract.md`

## 2. Легенда статусов
- `Уже совпадает` — текущее состояние в целом соответствует целевой модели.
- `Частично совпадает` — основа уже есть, но не хватает части поведения, ограничений или UX.
- `Нужно доработать` — текущее состояние заметно расходится с целевой моделью.

## 4. Где самые важные расхождения
Самые значимые расхождения между целевой моделью и текущей реализацией сейчас такие:

1. ИИ-чат должен стать `pet-centric`, а сейчас он `user-centric`.
2. `shared_user` по целевой модели должен иметь доступ к ИИ-чату питомца, а сейчас этого нет.
3. PWA/offline/store strategy зафиксирована, но сама frontend-реализация еще отсутствует.
4. Административная защита для `upload-data` должна быть усилена в коде.
5. Лимиты по размеру файлов пока не определены и не реализованы.
6. Геологика карты на уровне определения города пользователя пока не материализована в реализации.
7. Новый акцент на новичков, поездки и непредвиденные ситуации еще не полностью разложен по UX и логике продукта.

## 5. Что можно считать уже устойчивой основой проекта
Эти части уже выглядят как надежная база:
- backend-модульная структура;
- auth по email/password;
- backend support для Telegram auth;
- CRUD питомцев;
- upload/download flow для фото;
- upload/download flow для документов;
- sharing через invites;
- базовый map API;
- единый error format;
- комплект основной проектной документации.

## 6. Что лучше делать следующим
Практически самый разумный порядок доработок такой:

1. Доработать permission logic под целевую модель `shared_user`.
2. Спроектировать и реализовать `pet-centric` LLM chat.
3. Защитить `upload-data` через admin-only доступ.
4. Зафиксировать file size limits.
5. Разложить сценарии новичка, поездки и непредвиденной ситуации на конкретные frontend flows.
6. Начать frontend с учетом уже принятых продуктовых решений.

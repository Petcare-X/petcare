# Permission Matrix: PetCare

## 1. Назначение документа
Этот документ фиксирует роли и права доступа в проекте `PetCare`.

Он нужен для того, чтобы:
- синхронизировать продуктовые ожидания и текущую backend-реализацию;
- убрать двусмысленность для frontend, backend и ИИ-агентов;
- явно зафиксировать, какие действия разрешены каждой роли.

Связанные документы:
- продуктовые требования: `docs/PRD.md`
- frontend-архитектура: `docs/frontend-architecture.md`
- backend-архитектура: `docs/backend-architecture.md`

## 2. Статус документа
Текущее состояние:
- часть правил уже реализована в backend;
- часть правил является зафиксированной продуктовой целью, но еще не полностью реализована;
- в случаях расхождения документ явно различает `Текущее состояние backend` и `Целевая модель`.

## 3. Роли

### 3.1 `guest`
Неавторизованный пользователь.

### 3.2 `authenticated_user`
Обычный авторизованный пользователь без отношения к конкретному питомцу.

### 3.3 `pet_owner`
Авторизованный пользователь, который является владельцем конкретного питомца.

### 3.4 `shared_user`
Авторизованный пользователь, которому владелец предоставил доступ к конкретному питомцу.

### 3.5 `admin`
Привилегированная роль для административных операций, связанных со справочными данными и импортом CSV.

Примечание:
- формальная роль `admin` пока не отражена в текущем backend-коде как завершенная permission model, но уже зафиксирована как необходимая для административных сценариев.

## 4. Области доступа
Матрица покрывает следующие функциональные зоны:
- auth и пользовательская сессия;
- профиль пользователя;
- питомцы;
- фото питомца;
- документы питомца;
- sharing и invites;
- ИИ-чат;
- карта и рекомендации;
- административные операции с CSV и картографическими данными.

## 6. Правила по зонам

### 6.1 Auth и профиль пользователя
Правила:
- `guest` может только зарегистрироваться и войти;
- любой авторизованный пользователь может управлять своей сессией и собственным профилем;
- login поддерживает `email/password` и `Telegram`.

### 6.2 Питомцы
Правила:
- любой авторизованный пользователь может создать питомца;
- `pet_owner` имеет полный контроль над своим питомцем;
- `shared_user` может просматривать карточку питомца, но не редактировать и не удалять ее;
- список питомцев для авторизованного пользователя может включать как собственных, так и расшаренных питомцев.

### 6.3 Фото и документы
Правила:
- загружать, изменять и удалять файлы может только владелец питомца;
- `shared_user` имеет доступ ко всем документам питомца без дополнительной фильтрации;
- в первой версии поддерживаются форматы документов `PDF`, `JPG`, `PNG`;
- превью документов внутри приложения не является обязательным, базовый сценарий — скачивание.

### 6.4 Sharing
Правила:
- создавать и отзывать доступ может только владелец питомца;
- `shared_user` не может управлять шарингом;
- владелец не обязан видеть историю активных и истекших приглашений как отдельную обязательную функцию первой версии;
- принятие приглашения доступно авторизованному пользователю.

Примечание:
- просмотр списка пользователей с доступом для `shared_user` в продуктовой документации явно не зафиксирован как обязательная возможность, поэтому в матрице указан `Частично`.
- если потребуется, это лучше уточнить отдельно.

### 6.5 ИИ-чат
Целевая модель:
- чат привязан к конкретному питомцу;
- `pet_owner` может пользоваться ИИ-чатом питомца;
- `shared_user` тоже может пользоваться ИИ-чатом питомца;
- данные профиля питомца подключаются как контекст только по явному выбору пользователя;
- доступ ИИ к документам питомца пока не является обязательным сценарием.

Текущее состояние backend:
- чат сейчас привязан к пользователю, а не к питомцу;
- backend currently allows only owner-user chat ownership;
- права `shared_user` для LLM-чата еще не реализованы.

Следствие:
- это один из ключевых разрывов между текущей backend-реализацией и целевой продуктовой моделью.

### 6.6 Карта и рекомендации
Правила:
- карту могут видеть все пользователи, включая `guest`, если команда не решит ужесточить доступ позже;
- рекомендации и pet-related контекст актуальны для `pet_owner` и `shared_user`;
- карта в MVP должна поддерживать каталог точек, фильтры и определение города пользователя по геоданным;
- использование точного местоположения пользователя не является обязательным.

### 6.7 Административные операции
Правила:
- операции с CSV и картографическими данными должны быть полностью защищены;
- такие действия доступны только `admin`;
- обычные пользователи, владельцы питомцев и `shared_user` не должны иметь доступ к `upload-data` сценариям.

## 7. Текущее состояние backend vs целевая модель

### 7.1 Что уже в целом совпадает
- `pet_owner` управляет своим питомцем;
- `shared_user` может читать карточку питомца;
- `shared_user` может читать документы питомца;
- создание и отзыв шаринга доступны владельцу;
- административный доступ к CSV уже признан необходимым на уровне документации.

### 7.2 Что требует доработки backend
- `shared_user` пока не имеет целевых прав на ИИ-чат;
- ИИ-чат пока user-centric, а не pet-centric;
- административная защита `upload-data` должна быть явно реализована;
- формальная permission model для `admin` пока не оформлена;
- permission matrix еще не материализована как кодовая политика на всех эндпоинтах.

## 8. Правила для frontend
- Интерфейс должен скрывать действия, которые недоступны текущей роли.
- `shared_user` должен видеть read-mostly сценарий без действий редактирования и без управления шарингом.
- Frontend не должен полагаться только на UI-ограничения: backend остается источником истины по permission checks.
- Все клиентские guards должны строиться на этой матрице, но окончательное решение остается за сервером.

## 9. Правила для backend
- Все права должны проверяться на сервере вне зависимости от поведения клиента.
- Для каждой чувствительной операции должен быть явно определен субъект доступа.
- Административные маршруты должны быть отделены от пользовательских по модели доступа.
- При расширении ролей или сценариев сначала должен обновляться этот документ, затем API и реализация.

## 10. Открытые точки
- Нужно ли `shared_user` видеть список пользователей, которым тоже открыт доступ к питомцу.
- Нужна ли в будущем более детальная рольовая модель, например `editor`, `viewer`, `vet`, `moderator`.
- Нужно ли в будущем ограничивать `shared_user` доступ к отдельным разделам рекомендаций или аналитики.
- Нужно ли отделять `admin` и `internal operator` как разные роли.

## 11. Рекомендации по следующему шагу
- Сверить эту матрицу с текущими backend endpoints и сервисами.
- После этого создать `docs/api-contract.md`, уже опираясь на зафиксированные роли и права.
- Для backend-задач оформить отдельный список доработок permission logic.
